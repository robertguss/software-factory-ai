# Conveyor — Phase 2: Plan Compiler & Contract Foundry

> **Status:** brainstorming draft; not yet committed for implementation.
>
> **Purpose:** synthesize the living strategy, the Phase 0/1 implementation
> plan, and all three advanced-capability documents into the next coherent body
> of work after Phase 0/1. This is intentionally opinionated. It preserves the
> long-term ambition while resisting the temptation to build the fleet,
> autonomy machinery, or a planning IDE before the plan-to-contract boundary is
> proven.
>
> **Working product name for the phase:** **The Contract Foundry**.
>
> **One-line outcome:** a human supplies a finished plan; Conveyor returns a
> critic-reviewed, test-bearing, dependency-ordered, human-approved executable
> work graph whose Slices can be run by the existing Phase-1 loop without manual
> contract authoring.

---

## 0. Executive recommendation

The next implementation should be **Phase 2: Plan Compiler & Contract Foundry**,
preceded by a short **Phase-1 qualification checkpoint** using several real
Slices.

Do **not** go directly from one sterile FastAPI tracer Slice to a parallel fleet.
Do **not** attempt to implement every capability assigned to “Phase 2” across
all advanced documents. That would turn the most important trust boundary in
Conveyor into a sprawling platform project.

The immediate sequence should be:

1. **Qualify the Phase-1 loop on a small real corpus.** Run the manual
   Plan/Brief/TestPack workflow on multiple non-toy Slices and record schema,
   gate, artifact, and operator friction.
2. **Add immutable plan revisioning and a planning execution capsule.** Agentic
   planning work must be as reproducible and auditable as implementation work.
3. **Build a deterministic plan compiler around stochastic proposal agents.**
   Agents may propose questions, decompositions, contracts, tests, and critiques;
   only the deterministic compiler may materialize execution truth.
4. **Build the Contract Foundry.** Separate Decomposer, Test Architect, and
   Contract Critic roles create and pressure-test each Slice contract.
5. **Present one approval package.** The human reviews all agent-inferred
   assumptions, graph structure, contracts, tests, risks, and blockers in a
   single checkpoint, then approves by digest.
6. **Prove the output by executing generated Slices sequentially through the
   existing Phase-1 loop.** “The compiler produced valid JSON” is not success;
   generated contracts must survive real implementation and verification.

### 0.1 Why this is the correct next phase

The Phase-1 loop proves that Conveyor can execute **one already-good work
packet**. The next unknown is not concurrency. It is whether Conveyor can
reliably manufacture good work packets from a human plan.

That is the highest-leverage uncertainty because every later subsystem assumes
it:

- the Dispatcher assumes Slices are correctly sized and dependency-ordered;
- the WorkerPool assumes contracts are independently executable;
- the verification pyramid assumes acceptance criteria and tests are honest;
- the merge queue assumes interfaces and scope are explicit;
- self-healing assumes a failure can be attributed to plan, context, execution,
  or gate rather than to an incoherent contract;
- learning assumes the ledger labels work consistently enough to compare runs.

Parallelism before contract quality would simply multiply ambiguity.

### 0.2 Four corrections to the source plans

#### Correction A — capability IDs are currently ambiguous

The two later advanced documents both use **C11–C20 for different features**.
That is already unsafe for implementation tickets, schemas, ADRs, and commit
messages. Before Phase 2 begins, create a canonical capability registry and use
qualified aliases such as `V2-C14` and `EXP-C14` only for historical references.
This draft uses descriptive names and new phase-local IDs instead of the
colliding numbers.

#### Correction B — universal mutation testing at contract lock is circular

A conventional code mutation score requires a working implementation. Before an
implementer exists, a “reference solution” generated by the spec agent would
quietly couple the Test Architect to an implementation and weaken separation of
duties. Phase 2 should therefore hard-gate **calibration and integrity**, while
running code mutation only when a legitimate reference implementation exists.
Full mutation testing belongs after a candidate implementation or in the Phase-4
verification pyramid.

Phase 2 can still strengthen contracts through:

- red/green calibration with expected failure reasons;
- hermeticity and repeatability checks;
- adversarial “how could an implementation cheat this contract?” review;
- positive, negative, boundary, and non-goal examples;
- contract/test mutation where the contract oracle itself is perturbed;
- conventional code mutation in supported cases with a real reference.

#### Correction C — a changed RunSpec cannot resume the same RunAttempt

One advanced proposal allows micro-negotiation to create a new ContractLock and
RunSpec while resuming the same attempt. That conflicts with the Phase-0/1
invariant that a RunAttempt has one immutable RunSpec. Phase 2 should instead:

- resolve contract negotiation **before** an execution attempt starts; or
- create a new RunAttempt after an amendment, while recording that it did not
  consume an implementation-failure retry budget.

No attempt may change its contract in place.

#### Correction D — the Phase-2 feature list must be cut aggressively

A full Workbench, Evidence Time Machine, autonomous triage, mutation framework,
plan simulator, micro-negotiation engine, and decomposition system in one phase
would obscure the actual experiment. Phase 2 should ship thin versions only
where they directly support plan compilation and approval.

## 1. Phase-2 product contract

### 1.1 Public promise

> **Conveyor compiles a human-authored plan into an inspectable executable work
> graph containing bounded Slices, explicit dependencies, locked contracts,
> independently authored tests, and a complete approval bundle. A human approves
> the graph once; approved root Slices can then enter the existing verified
> execution loop.**

### 1.2 What the human still owns

The human remains the author of product intent, priorities, architecture taste,
non-goals, and material trade-offs. External research and multi-model planning
remain outside Conveyor for this phase.

The human:

- supplies the finished plan and repository;
- answers one consolidated clarification batch when necessary;
- reviews the executable graph and agent-inferred assumptions;
- approves or rejects each Epic within one approval checkpoint;
- decides material plan amendments.

### 1.3 What Conveyor owns in Phase 2

Conveyor owns:

- plan revisioning and normalized-plan compilation;
- deterministic and agentic specification interrogation;
- repository planning context;
- Epic/Slice decomposition proposals;
- dependency and interface modeling;
- Agent Brief and DiffPolicy drafting;
- independent Test Architect output;
- test calibration and integrity checks;
- adversarial contract criticism;
- bounded repair loops;
- the canonical approval bundle;
- contract locking and publication of the ready pool;
- traceability from plan text to every generated artifact;
- controlled amendment proposals after approval.

### 1.4 Non-goals

Phase 2 does **not** build:

- parallel execution, WorkerPool, Dispatcher, merge queue, or credential pool;
- auto-merge or auto-deploy;
- full epic/phase verification pyramid;
- cost-calibrated swarm simulation;
- adaptive model routing or agent reputation;
- autonomous micro-negotiation of contract changes;
- self-healing retries or supervisor agents;
- a rich drag-and-drop planning IDE;
- general brownfield characterization;
- a standalone PR-review product;
- full event-sourced domain reconstruction;
- universal multi-language mutation testing.

### 1.5 Definition of done

Phase 2 is complete when Conveyor can take a multi-Epic plan and:

1. create an immutable `PlanRevision` and `PlanningSpec`;
2. produce one consolidated clarification batch, with every agent-inferred
   assumption made visible;
3. compile a schema-valid, acyclic work graph with typed dependency edges;
4. create Agent Briefs, structured interfaces, DiffPolicies, risks,
   archetypes, acceptance criteria, and test specifications for every Slice;
5. have a separate Test Architect create locked TestPacks where executable tests
   are appropriate;
6. calibrate every TestPack against the base repository and reject invalid,
   flaky, non-hermetic, unexpectedly green, or malformed packs according to
   policy;
7. have a separate Contract Critic attempt to exploit underspecification and
   return structured findings;
8. produce a canonical planning bundle and a read-only approval Workbench;
9. record human approval against the exact bundle root digest;
10. lock approved contracts and publish dependency-ready root Slices;
11. execute a representative set of generated Slices sequentially through the
    Phase-1 loop with no manual rewriting of their contracts;
12. support a material plan amendment through a new plan revision, impact
    analysis, re-approval, and new locks without mutating historical evidence.

## 2. Phase-1 exit qualification: the work before new machinery

The current Phase-1 definition proves one deliberately clean tracer. That is not
enough data to freeze the contract compiler around.

### 2.1 Qualification corpus

Before implementing the agentic decomposer, run **at least six manually authored
Slices** through the Phase-1 loop:

- at least two feature additions;
- at least one bug fix;
- at least one refactor/no-behavior-change Slice;
- at least one Slice touching a public interface;
- at least one Slice that fails or becomes disputed for a legitimate reason;
- across at least two repositories, including one real Conveyor-adjacent repo;
- preferably across at least two toolchain profiles, without making broad
  multi-language support a release requirement.

The goal is not statistically significant model evaluation. The goal is to find
schema and workflow assumptions that one toy Slice cannot reveal.

### 2.2 Required closeout report

Produce `phase1-qualification.md` containing:

- fields repeatedly missing from Agent Briefs;
- contract changes requested after implementation began;
- test calibration failures and causes;
- context-pack misses;
- evidence artifacts reviewers actually used;
- operator steps that felt redundant or confusing;
- state-machine transitions that did not match reality;
- adapter limitations encountered;
- gate false positives and false negatives;
- average human time to author a Slice contract;
- an explicit “safe to compile automatically?” decision.

### 2.3 Schema-seam freeze audit

Before Phase 2 migrations, verify or add the cheap historical-data seams that
are genuinely valuable:

**Required:**

- stable canary mutant identity and origin;
- `findings[].rule_key`;
- `contract_disputed` off-ramp;
- TestPack strength/integrity status fields;
- stable `archetype_key` and run cost/duration fields;
- structured interface keys;
- `Slice.stable_key` and explicit dependency-edge semantics;
- context-usage telemetry when the adapter can reliably derive it.

**Optional, safe to add later:**

- plan-graph projection reference;
- in-loop tutor iteration fields;
- authorized-scope fields if Phase 2’s contract schema can add them cleanly.

Do not implement any advanced mechanism in this checkpoint. This is only schema
shape and evidence preservation.

## 3. Phase-2 design laws

1. **Agents propose; the compiler materializes.** No planning agent writes
   directly to execution resources or changes state.
2. **No hidden inference.** Every generated fact not copied from an authoritative
   human source is tagged as inferred, with provenance and confidence.
3. **No plan mutation in place.** Any meaningful change creates a new immutable
   `PlanRevision`.
4. **No approval without a digest.** Human approval binds to one canonical
   planning-bundle root digest.
5. **No final IDs from models.** Agents may use local labels; the deterministic
   compiler assigns and preserves canonical identities.
6. **No orphan requirement, AC, test, Slice, interface, or dependency edge.**
7. **No contract without a verification strategy.** A Slice lacking an honest
   oracle is split, clarified, explicitly made human-verified, or rejected.
8. **No self-authored acceptance authority.** Decomposer, Test Architect,
   Contract Critic, implementer, and execution reviewer are distinct roles.
9. **No fake certainty.** Low-confidence inferred facts are visible and can
   block approval based on risk.
10. **No infinite repair loops.** Each stochastic station has a bounded repair
    budget and deterministic terminal outcome.
11. **No interface over-freezing.** Only public or cross-Slice interfaces are
    contractually locked by default; internal implementation choices remain free
    unless a human architectural decision says otherwise.
12. **No circular test-strength proof.** A model-generated hidden reference
    implementation is not accepted as the universal basis for contract mutation
    quality.
13. **No graph edge without semantics.** Dependencies must state why they exist
    and whether they block execution, integration, or verification.
14. **No Phase-3 leakage.** Structural previews are allowed; a real scheduler,
    fleet, merge queue, and cost simulator are not.

## 4. Architecture overview

```text
Human Plan
   │
   ▼
Immutable PlanRevision + PlanningSpec
   │
   ├── Deterministic Plan Audit
   ├── Spec Interrogator → one question batch
   ├── Human Decisions / accepted defaults
   ├── Planning Context Scout
   ├── Decomposer proposal
   ├── Deterministic Work-Graph Compiler
   ├── Contract Forge
   ├── Independent Test Architect
   ├── Calibration + Integrity checks
   ├── Adversarial Contract Critic
   ├── Bounded repair loop
   ├── Structural graph dry-run
   └── PlanningBundle / Approval Workbench
                         │
                         ▼
                    Human approval
                         │
                         ▼
      Locked AgentBriefs + TestPacks + ContractLocks
                         │
                         ▼
          Approved root Slices enter Phase-1 RunSlice loop
```

The architecture is a compiler pipeline, not a chat among agents. Every
stochastic station emits a typed proposal artifact. Deterministic code validates,
normalizes, assigns identities, computes digests, and decides whether the next
station is allowed to run.

## 6. Field-level provenance: the Inference Ledger

This should be a first-class Phase-2 concept.

The main human trust problem is not merely “what did the model output?” It is
“which parts of this executable graph came directly from my plan, which were
inferred from the repository, and which were invented as a proposed default?”

Every meaningful generated value should accept a provenance envelope:

```elixir
%{
  origin: :human_explicit | :human_decision | :repo_observed |
          :agent_inferred | :deterministic_derived,
  source_refs: ["plan.md#REQ-004", "app/routes.py:21-58"],
  confidence: :high | :medium | :low,
  inference_reason: nil | "Route and schema changes appear inseparable",
  approval_status: :not_required | :pending | :accepted | :rejected
}
```

The approval Workbench defaults to an **inference-first review mode**:

- show low-confidence or high-impact inferred facts first;
- collapse facts copied directly from the human plan;
- highlight inferred dependencies, interfaces, non-goals, risks, and test
  oracles;
- require explicit human acceptance for high-impact inferred facts.

No hidden assumptions survive approval.

### 6.1 Assumption budget

The compiler maintains an embedded assumption register:

```text
key
statement
affected_refs[]
impact ∈ low | medium | high
confidence
proposed_default?
resolution ∈ unresolved | accepted_default | human_decision | rejected
```

Policy examples:

- any unresolved high-impact assumption blocks approval;
- a Plan may contain at most N unresolved low-impact defaults;
- a public-interface assumption always requires explicit acceptance;
- accepted defaults become `HumanDecision` records at approval.

This is more useful than a vague readiness percentage because it exposes the
actual uncertainty the human is accepting.

## 7. The Phase-2 station pipeline

### P2-S1 — Ingest immutable plan revision

Inputs:

- source Markdown and `conveyor.plan@1` block/sidecar;
- repository identity and base commit;
- human decisions already attached to the Plan.

Outputs:

- `PlanRevision`;
- normalized plan artifact;
- canonical source-span map;
- `PlanningSpec` and StationPlan.

Acceptance:

- revision is immutable;
- all source references resolve;
- same canonical input produces the same contract digest;
- unknown schemas fail explicitly.

### P2-S2 — Deterministic plan audit and interrogation

Extend the Phase-1 PlanAudit with high-precision checks:

- missing or orphan requirements/ACs;
- undefined references;
- unmeasurable acceptance language;
- contradictory enum values, status codes, or interface claims;
- requirements that mix unrelated risk domains;
- missing non-goals;
- missing human decisions for protected architectural choices;
- missing test oracle;
- suspiciously broad “do everything” requirements;
- source plan instructions that conflict with Conveyor policy.

Deterministic findings are authoritative blockers.

### P2-S3 — Agentic Spec Interrogator

A separate, read-only interrogator profile receives:

- normalized plan;
- deterministic findings;
- human decisions;
- policy and question-output schema.

It returns one deduplicated question batch containing:

- ambiguity;
- contradiction;
- untestable requirement;
- hidden dependency;
- missing decision;
- non-goal collision;
- unsafe implied behavior;
- proposed default where appropriate.

The interrogator cannot edit the plan. It asks only.

Rules:

- one batch per revision unless an answer creates a genuinely new conflict;
- hard questions block decomposition;
- soft questions may carry proposed defaults;
- every question states why it matters and what downstream failure it prevents;
- false-alarm feedback is recorded for prompt/eval improvement.

### P2-S4 — Resolve questions into a new revision

Human answers become `HumanDecision` records. Accepting proposed defaults is a
human decision, not silent model authority.

Any answer changing the normalized contract creates a new `PlanRevision` and
new `PlanningSpec`. The prior interrogation remains historical evidence.

### P2-S5 — Planning Context Scout

Before decomposition, build a repository-level planning context artifact.
This is broader than the per-Slice ContextPack.

Contents:

- architecture/module map;
- public interfaces and schemas;
- existing test topology and test commands;
- package/dependency boundaries;
- migrations and persistence model;
- ADRs and project instructions;
- CodeScent or local quality hotspots;
- likely protected paths;
- known conventions and naming patterns;
- dynamic-language uncertainty notes;
- citations for every observation.

The station is read-only. Start deterministic (`rg`, manifests, route/schema
extractors, CodeScent) and allow an optional read-only planning-scout agent.

### P2-S6 — Decomposer proposal

The Decomposer receives:

- approved normalized PlanRevision;
- planning context;
- decomposition policy;
- controlled Slice archetype vocabulary;
- required output schema.

It proposes:

- Epics;
- Slices;
- requirement and decision coverage;
- typed dependency edges;
- likely files and conflict domains;
- risk and autonomy ceiling;
- structured provided/required interfaces;
- preliminary acceptance criteria;
- non-goals and authorized scope;
- unresolved assumptions;
- rationale for every Slice boundary.

The proposal is an artifact only. It does not create Ash work records.

### P2-S7 — Deterministic work-graph compiler

The compiler:

1. validates the proposal schema;
2. resolves every source reference;
3. assigns canonical stable keys and internal IDs;
4. checks complete requirement and AC coverage;
5. detects cycles and duplicate/conflicting edges;
6. validates dependency semantics;
7. checks public/cross-Slice interface ownership;
8. rejects impossible or orphan Slices;
9. applies size and scope heuristics;
10. verifies no model-authored field is missing provenance;
11. materializes draft Epics, Slices, Agent Briefs, and dependencies in one
    transaction only after all structural checks pass;
12. emits deterministic diagnostics for repair.

#### Canonical identity policy

Agents never assign final IDs. The compiler assigns stable keys and preserves
identity through explicit `supersedes` relations. Reordering a proposal may not
renumber unrelated Slices.

#### Slice size checks

A Slice is suspicious when it:

- spans multiple unrelated primary behaviors;
- crosses unrelated risk domains;
- has no independently checkable oracle;
- has an unbounded likely-file set;
- requires several unrelated public interface changes;
- cannot produce a complete RunPrompt without placeholders;
- has acceptance criteria that depend on future Epics rather than declared
  dependencies.

These are findings, not simplistic line-count truth. High-risk or ambiguous
cases require split/review.

### P2-S8 — Structural graph dry-run

Before authoring tests, deterministically simulate graph progression:

- topologically traverse the graph;
- calculate execution waves;
- prove every Slice can eventually become ready;
- distinguish execution blockers from integration/verification relationships;
- identify maximum parallel width for future Phase 3;
- report likely-file and conflict-domain collisions;
- report high-fan-out dependencies and single points of failure;
- compute a structural critical path by edge count/risk, **not** fake time/cost
  estimates.

This is not the future calibrated swarm simulator. It is a graph correctness and
approval aid.

### P2-S9 — Contract Forge

For each Slice, a contract-authoring pass produces a full Agent Brief.
The Decomposer may seed it, but the Contract Forge normalizes it against the
actual contract schema.

Every contract includes:

- current and desired behavior;
- source requirements and decisions;
- structured interfaces;
- acceptance criteria;
- positive, negative, boundary, and non-goal examples;
- properties/invariants where appropriate;
- required tests and verification suites;
- authorized scope and protected paths;
- risk and required review kinds;
- likely files and conflict domains as hints, not implementation commands;
- explicit assumptions;
- done definition;
- out-of-scope behavior;
- provenance on every inferred field.

#### Interface lock levels

```text
strict               exact public/cross-Slice shape is locked
compatible_superset  additive compatible changes allowed
review_required      changes permitted only through amendment/review
informational        internal implementation hint; not a lock
```

Default to strict only for genuinely public or cross-Slice interfaces.

### P2-S10 — Independent Test Architect

A Test Architect profile distinct from Decomposer, Critic, and implementer
receives:

- normalized plan and Slice contract;
- read-only planning context;
- repository base commit;
- test-author policy;
- write access only to a test-pack workspace.

It produces:

- `TestSpecification` artifact;
- required test IDs;
- test roles;
- oracle definitions;
- executable TestPack patch when supported;
- property generators or example tables where appropriate;
- expected base behavior and expected patched behavior;
- runner commands and result adapters.

#### Test roles

```text
acceptance_new          expected to fail on base for a specific missing behavior
bug_reproduction        expected to fail on base for the reported defect
regression_preservation expected to pass on base and patch
characterization        records current behavior; does not claim correctness
property                checks an invariant over generated cases
interface_contract      verifies a public/cross-Slice interface
security_policy         verifies a policy/security condition
human_verification      explicit manual oracle; blocks higher autonomy
```

This prevents the simplistic assumption that every new locked test must be red
on base.

### P2-S11 — Calibration and test integrity

Hard Phase-2 checks:

1. Test IDs resolve and map to ACs.
2. Baseline-preservation tests pass on base.
3. New-behavior/bug tests fail on base for an expected reason, not import or
   infrastructure breakage.
4. Repeated executions produce a stable result/failure signature.
5. Tests obey sandbox/network/time/RNG policy.
6. The TestPack cannot edit or escape its mount.
7. Required commands and structured result artifacts exist.
8. No test weakens policy or edits production code.
9. Every locked public interface has a declared test oracle, even when dynamic
   coverage cannot yet prove execution.

Integrity status dimensions:

- calibration;
- hermeticity;
- repeatability/flake;
- red-on-stub where a supported honest stub can be generated;
- interface-oracle coverage;
- contract strength assessment.

#### What is hard-blocking in Phase 2

Hard-block:

- malformed or missing tests;
- unexpected green on a required red case;
- wrong failure reason;
- flaky or non-hermetic required test;
- vacuous test detected by a supported sentinel;
- missing AC mapping;
- hidden network or secret dependency;
- test-author/implementer role collision.

Advisory until calibrated:

- conventional code mutation score without a real reference implementation;
- dynamic interface coverage for not-yet-existing code;
- heuristic assertion-strength scores;
- unsupported-language stub analysis.

### P2-S12 — Adversarial Contract Critic

A separate critic profile reads only the planning bundle, not hidden generation
reasoning.

Its primary question is:

> “What is the cheapest wrong implementation that could satisfy these written
> acceptance criteria and tests while violating the human’s actual intent?”

Required critic lenses:

- requirement coverage;
- internal contradiction;
- scope ambiguity;
- interface compatibility;
- dependency correctness;
- test oracle weakness;
- missing negative/boundary cases;
- likely spec-gaming implementation;
- risk under-classification;
- non-goal leakage;
- hidden human decision.

Output is schema-valid findings with stable rule keys, evidence refs, and
concrete proposed repairs. The critic cannot approve or lock contracts.

### P2-S13 — Bounded repair loop

For invalid proposals or critic findings:

- deterministic diagnostics are fed to the responsible authoring role;
- at most two automatic repair rounds per station by default;
- the generator may change only the rejected artifact scope;
- every revision gets a new digest and comparison artifact;
- repeated non-progress or oscillation stops and routes to the human;
- material plan changes cannot be repaired silently; they become an amendment
  or clarification decision.

### P2-S14 — Prompt dry-compile

Before approval, run the existing PromptBuilder in dry mode for every Slice.
A contract is incomplete if Conveyor cannot render a valid future RunPrompt
without placeholders or unresolved references.

Validate:

- Brief, policy, interfaces, tests, and output schema are present;
- no instruction hierarchy conflict;
- referenced artifacts exist;
- planned autonomy does not exceed adapter/sandbox capability;
- the prompt remains within configured size bounds;
- every untrusted excerpt is labeled.

No implementer is launched.

### P2-S15 — Build canonical approval bundle

The bundle contains:

- normalized PlanRevision and human decisions;
- question/answer history;
- field-level inference report;
- graph and structural dry-run;
- requirement/AC/Slice/test coverage matrices;
- all Agent Briefs and TestSpecifications;
- TestPack calibration and integrity results;
- critic findings and repairs;
- risk/approval summary;
- contract and plan diffs from the previous revision;
- unresolved warnings;
- planning prompts, profiles, policy, and provenance digests.

Approval is impossible until the bundle root digest is stable.

### P2-S16 — Human approval checkpoint

One checkpoint, with Epic-level granularity.

The human can:

- approve an Epic;
- reject an Epic;
- request a Slice split/merge;
- change an AC, non-goal, risk, dependency, or interface lock;
- accept or reject an inferred assumption;
- mark a Slice as human-verification-only;
- defer a requirement explicitly.

Edits never mutate approved truth directly. They create a structured change set,
new PlanRevision where needed, and rerun affected compiler/audit stages.

When all required Epics are approved, `HumanApproval` records:

- approval bundle root digest;
- approved Epic/Slice digests;
- actor and rationale;
- accepted warnings/assumptions;
- autonomy ceiling.

### P2-S17 — Lock and publish ready pool

On approval:

- create final ContractLocks;
- lock TestPacks;
- transition draft Slices to approved;
- mark dependency-free, readiness-clean roots as ready;
- keep blocked descendants approved until dependencies are done;
- project the approved plan bundle;
- emit ledger events and LiveView updates.

A simple deterministic `next_ready` query is allowed. A Dispatcher is not.

### P2-S18 — Sequential execution validation

Select representative approved Slices and run them one at a time through the
existing Phase-1 loop.

This is the compiler’s integration test. Track:

- first-pass gate success;
- implementer clarification/dispute rate;
- context-pack misses;
- missing tests/interfaces;
- contract amendments after execution starts;
- human edits required to make a generated contract runnable.

A Phase-2 release cannot be declared solely from planning artifacts.

```json
      "stable_key": "SLC-TASK-FILTER-7F3A",
      "title": "Add completed-state filtering",
      "archetype_key": "crud_query_filter",
      "source_refs": ["REQ-014", "AC-021"],
      "risk": "low",
      "autonomy_ceiling": "L1",
      "likely_files": ["app/routes.py", "app/repository.py"],
      "conflict_domains": ["tasks_api", "task_query"],
      "provided_interfaces": [],
      "required_interfaces": [],
      "authorized_change_globs": ["app/**", "tests/**"],
      "provenance": {}
    }
  ],
  "dependencies": [
    {
      "from": "SLC-SCHEMA-19A2",
      "to": "SLC-TASK-FILTER-7F3A",

      "interface_keys": ["db.tasks.completed"],
      "rationale": "Filter query requires the persisted completed column",
      "provenance": {}
    }
  ]
}
```

### 8.1 Dependency semantics

Avoid treating every relationship as `blockedBy`.

- `execution_hard`: successor cannot be implemented meaningfully first.
- `interface`: successor depends on an interface; treated as hard in Phase 2,
  later eligible for stub parallelism.
- `integration_order`: implementation may proceed, but merge order matters.
- `verification`: both may execute, but a combined gate waits for both.
- `human_decision`: work blocks on an unresolved decision.
- conflict domains and likely files are **scheduling hints**, never dependency
  edges by themselves.

This distinction prevents needless serialization and preserves the future
throughput path without building it now.

## 10. Human approval Workbench

- blockers;
- high-impact assumptions;

Everything available in LiveView must be available in

## 11. Plan amendments and contract disputes

### 11.1 Material amendment path

A material proposal:

1. records evidence and affected refs;
2. moves the affected Slice to `contract_disputed` or keeps it unready;
3. computes affected and downstream Slices;
4. creates a proposed redline;
5. requires a HumanDecision;
6. creates a new PlanRevision when accepted;
7. recompiles and re-audits the affected graph;
8. creates new ContractLocks and future RunAttempts;
9. leaves all historical evidence interpretable against old locks.

### 11.2 Phase-2 position on micro-negotiation

Do not auto-adjudicate contract changes yet.

Collect structured dispute data first:

- request kind;
- proposed delta;
- deterministic materiality classification;
- human decision;
- whether the request was legitimate;
- whether it would have been safe to automate.

After enough evidence, a later phase may auto-accept strict interface supersets
or clarifications. Phase 2 keeps every contract change human-approved.

### 11.3 No retry penalty for contract faults

When an implementation discovers an impossible contract:

- classify it as a contract/plan fault, not an implementation failure;
- do not consume the implementer’s rework budget;
- terminate the immutable attempt cleanly;
- route through amendment;
- start a new attempt under the new RunSpec after approval.

## 15. Safety and threat model additions

Threats specific to Phase 2:

- plan text attempts to instruct planning agents to bypass policy;
- repository docs/comments poison decomposition;
- Decomposer invents requirements not present in intent;
- Test Architect encodes a narrower behavior than the AC;
- generator and critic collude through shared profile/context;
- human approves a bundle different from the one later locked;
- repair loop silently weakens acceptance;
- model-generated IDs collide or drift;
- a test-author workspace mutates production source;
- an amendment invalidates downstream contracts without detection.

Defenses:

- trust labels on all plan/repository/tool excerpts;
- field-level provenance and inference-first approval;
- separate profiles and actor policies;
- deterministic compiler and identity assignment;
- bundle-root approval binding;
- contract-diff classification with acceptance weakening blocked;
- read-only mounts and test-only write roots;
- bounded repair loops;
- impact analysis and semantic invalidation;
- prompt-injection fixtures for every planning role.

## 17. Phase-2 KPIs and release thresholds

### 17.1 Hard correctness thresholds

- 100% requirement-to-AC-to-Slice traceability for approved scope.
- 0 orphan Slices, tests, interfaces, or dependency edges.
- 0 cycles in executable dependency graph.
- 100% approved bundles reproducible by root digest.
- 100% locked contracts tied to an explicit HumanApproval.
- 0 contract mutation in place.
- 0 role-separation violations.
- 0 enabled planning prompt-injection fixture escapes.
- 0 required test packs with unexplained calibration or flake failures.

### 17.2 Initial quality targets

Treat these as hypotheses to calibrate, not marketing promises:

- at least 80% of generated Slices approved without being rewritten from
  scratch;
- median no more than one bounded contract-repair round per Slice;
- at least 70% first-pass deterministic gate success on the Phase-2 sequential
  execution corpus;
- material contract-dispute rate below 20% of executed generated Slices;
- no more than one consolidated human clarification batch per plan revision;
- human approval time under 30 minutes for a 10-Slice plan after familiarity;
- interrogation hard-finding precision above 80%; false alarms are measured;
- 100% of high-impact inferred assumptions explicitly accepted or rejected.

The Phase-3 go/no-go decision should use observed data, not force these exact
numbers if the corpus proves they are poorly chosen.

## 20. Risks and mitigations

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Automating a bad manual schema | One tracer Slice may hide missing concepts | Phase-1 qualification corpus before compiler work |
| Agent decomposition looks plausible but is structurally wrong | Human may approve polished nonsense | deterministic compiler, provenance, structural dry-run, critic |

| Agentic interrogation annoys users | False questions reduce trust | deterministic checks first, precision metric, soft defaults, one batch |
| Phase 2 silently grows into Phase 3 | Scheduler concerns leak into compiler | structural dry-run only; explicit non-goals and cutline |

- **Contract criticism explicitly searches for the cheapest wrong
  implementation that could pass.**
- **Prompt dry-compilation proves every work packet can reach the existing
  execution loop.**
- **Mutation-at-lock is conditional, not circular theater.**
- **Contract changes create new revisions and attempts; nothing mutates in
  place.**
- **Generated plans prove themselves by executing real Slices sequentially
  before Conveyor earns a fleet.**

## 24. Additional high-leverage ideas considered

These are deliberately not all in the Phase-2 core. They are recorded so the
best ideas are not lost and can be promoted if qualification data justifies
them.

### 24.1 Static archetype contract templates — recommended for core

Use a small controlled vocabulary and minimum obligations rather than asking a
model to reinvent contract structure every time.

Examples:

| Archetype | Required contract obligations |
| --- | --- |
| `bugfix_regression` | reproduction test, cause hypothesis, unchanged-behavior list |
| `crud_endpoint` | request/response interface, negative status cases, persistence reflection |
| `schema_migration` | forward/backward compatibility, data preservation, rollback/irreversibility decision |
| `pure_refactor` | behavior-preservation oracle, public-interface freeze, no feature ACs |
| `dependency_update` | lockfile scope, advisory/security rationale, regression and rollback plan |
| `security_hardening` | threat scenario, abuse case, negative tests, mandatory security review |
| `performance` | workload, baseline, threshold, variance policy, regression budget |
| `configuration` | default, override precedence, invalid value behavior, secret handling |

Templates should be deterministic policy, not hidden prompt lore. A Slice can
use `custom`, but that increases approval scrutiny.

### 24.2 Anti-overdecomposition budget — recommended for core

The system must reject both giant Slices and confetti graphs.

Track:

- Slice count relative to requirements;
- edge density;
- average ACs per Slice;
- number of Slices touching the same likely files;
- expected fixed station overhead;
- graph fan-out/fan-in;
- number of Slices whose only independent oracle is the same integration test.

Findings:

- `slice_too_large`;
- `slice_too_small_to_justify_run`;
- `coordination_overhead_dominates`;
- `false_parallelism`;
- `shared_oracle_prevents_independent_verification`.

A good decomposition minimizes total expected work, not Slice size alone.

### 24.3 Atomicity groups — recommended seam

Some requirements should not be split even if they look separable: schema
change plus data backfill, authorization check plus audit log, or transaction
write plus outbox event.

Allow:

```text
atomicity_group_key
reason
members[]
policy ∈ same_slice | same_epic_gate | same_integration_batch
```

This prevents the compiler from manufacturing intermediate states that are
technically mergeable but operationally unsafe.

### 24.4 Contract falsifiability proof — recommended for critic rubric

For every AC, the Test Architect or Critic must provide at least one concrete
counterexample that would fail it. If no counterexample can be described, the
AC is likely aesthetic, tautological, or untestable.

### 24.5 Alternate-decomposition shadow pass — experimental

For high-risk plans, run a second independent Decomposer in shadow and compare:

- requirement coverage;
- Slice boundaries;
- dependency edges;
- public interface ownership;
- risk classification.

Do not merge outputs automatically. Surface material disagreements to the human.
This is useful as an eval/calibration tool before it becomes a production cost.

### 24.6 Slice coalescer/splitter — later Phase 2 optimization

After the first draft graph, a deterministic/agentic optimizer may suggest:

- merge Slices that share one oracle and the same files;
- split Slices that span risk domains or public interfaces;
- convert false hard dependencies into integration-order edges;
- group trivial mechanical changes to reduce station overhead.

It should propose a graph patch, never rewrite directly.

### 24.7 Approval cognitive-load budget — useful metric

Estimate review load from:

- number of inferred facts;
- high-risk/public interfaces;
- critic findings;
- unresolved warnings;
- contract length;
- change size from previous revision.

If the approval package exceeds policy, require more summarization or split the
approval into Epic tabs within the same checkpoint. Do not pretend a 200-Slice
bundle received meaningful human approval in one click.

### 24.8 Semantic scope delta detector — recommended

Compare source intent to generated graph and classify:

```text
scope_preserved
scope_added
scope_removed
scope_reinterpreted
unknown
```

Any `scope_added` item requires explicit approval and provenance. This catches a
model “helpfully” inventing features.

### 24.9 Contract challenge cases — recommended

Alongside ordinary examples, preserve a small `challenge_cases[]` list created
by the Critic. These are not necessarily executable tests yet; they are the
specific adversarial situations most likely to expose a too-weak contract.
They later seed red-team and mutation suites.

### 24.10 Interface consistency solver — later

For structured cross-Slice interfaces, validate that providers and consumers
agree on:

- names and versions;
- types/schemas;
- error behavior;
- compatibility mode;
- lifecycle/ownership.

A simple deterministic schema unifier is valuable before full stub generation.
Do not build a general theorem prover.

### 24.11 Human approval signatures — later trust upgrade

Phase 2 binds approval to a digest. A later incremental upgrade can sign the
approval statement using a local key or Sigstore-compatible identity. This is
useful for teams and remote runners, but not required for the local OSS tracer.

### 24.12 “Why this Slice?” capsule — recommended

Every Slice gets a tiny compiler-generated explanation:

- why it is separate;
- why it is independently verifiable;
- what it unlocks;
- why its dependencies are necessary;
- what would be lost by merging it with neighbors.

This dramatically improves human graph review and future scheduler debugging.

### 24.13 Decision-debt meter — later projection

Track accepted defaults and deferred human decisions that remain embedded in
active contracts. High-impact or aging decision debt can block later autonomy.
This is a more honest trust signal than merely counting approved plans.

### 24.14 Planning SARIF export — cheap product integration

Plan-audit and contract-audit findings can optionally export as SARIF so IDEs
and CI systems show source-linked plan problems. This is an adoption nicety, not
core Phase 2.

### 24.15 Contract lint as a standalone command — likely high ROI

Even users who do not adopt full execution may use:

```bash
mix conveyor.plan_prepare PLAN.md --no-agents
mix conveyor.contract_lint agent_brief.yml
```

A deterministic-only mode can become a small adoption wedge without creating a
separate product track yet.

### 24.16 Decomposition confidence calibration — later learning seed

Record Decomposer confidence per proposed edge/Slice, then compare it with human
edits and downstream disputes. Confidence is not authority, but calibration data
will show whether the model knows when it is guessing.

### 24.17 Contract provenance graph — later visualization

The inference ledger can become a graph showing exactly which plan paragraphs,
repo observations, human decisions, and agent inferences produced each
contract field. Useful for audits, but the Phase-2 Workbench can start with
source links and badges rather than a separate graph engine.

### 24.18 Approval shadow predictor — do not build yet

A model could predict which inferred items the human will reject and reorder the
Workbench. This is attractive but premature; the general Human Attention Queue
and learning loop are better homes after enough approval history exists.

## 25. Alternative sequencing considered

### Alternative A — go directly to Phase 3 fleet

**Rejected for now.** It delivers visible parallelism quickly, but scales an
unproven manual contract-authoring process and makes failures harder to
attribute. It is the best demo path and the wrong trust path.

### Alternative B — build the full verification pyramid next

**Tempting but incomplete.** A stronger gate helps, yet its quality remains
capped by plan and contract quality. The most valuable Phase-4 ideas should
influence Phase-2 schemas, but the full pyramid should follow the compiler.

### Alternative C — build a standalone PR reviewer next

**Strong product wedge, wrong immediate core sequence.** It could attract users
and harden the gate, but it does not advance the human-plan → executable-factory
loop. Keep as a post-Phase-4 parallel track.

### Alternative D — build brownfield onboarding next

**Strategically important, technically premature.** Real repositories are the
market, but characterization and behavior-lock machinery depend on a stable
verification pyramid. The Phase-1 qualification corpus should include real
repos without promising automated brownfield onboarding.

### Alternative E — build only a read-only Workbench next

**Too UI-heavy.** A Workbench without a trustworthy compiler is a polished view
of untrusted proposals. Build the canonical IR/compiler first; UI is a
projection.

### Alternative F — build only the Decomposer next

**Too stochastic.** A Decomposer without interrogation, test architecture,
criticism, and approval evidence merely moves manual ambiguity into generated
JSON. The smallest honest unit is the full Contract Foundry loop, with a strict
cutline around it.
