Grounding Context — Codebase

What exists today

The cockpit is /runs → CockpitLive: a live work-DAG drawn with Cytoscape + ELK in assets/js/hooks/dag.js. Nodes are 46px colored circles; styling is ~20 lines of hand-typed CSS inlined in the LiveView; the root layout is a bare <body> with no app shell, no design system, no dark mode. Founder's verdict: "looks awful." The gap to Stripe-level is not the graph library — it is the absence of a design system, an app shell, real node craft, and motion.

The locked stack (this ideation sits on top of it)

A uniform React frontend via Inertia.js on Phoenix — controllers return props → React pages; the server stays the authority. The one live screen (the cockpit) receives real-time deltas over a Phoenix Channel. Graph rendering moves to React Flow v12; the design layer is shadcn/ui + Radix + Tailwind + Motion + Lucide. One cohesive operator app, single deploy.

The data we can visualize (real, server-projected)

Nodes are slices with eight states — stalled, running, skipped, blocked, ready_idle, done, failed, parked. Each carries epic_id (parent grouping), blocked_by (unsatisfied upstream deps), and starved_dependents (count of downstream work waiting on it). Edges are dependencies. The server pushes graph:init then incremental node:patch deltas as runs progress; multiple runs exist, one viewed at a time.

The strategic insight from best-in-class research

"Premium" (Stripe / Linear / Vercel) is a craft level, not an aesthetic: near-mono palette + one accent, one type family, tabular numerals, generous whitespace, and every micro-state designed. The graph tools that read as products (GitHub Actions, Dagster, Prefect, n8n, Temporal, Inngest) render nodes as status cards, not circles. But the load-bearing warning: the mono-plus-whitespace-plus-one-accent look is becoming the generic dev-tool default — "serious, but says nothing about who you are." So the distinctive identity must live in the live DAG's node / edge / motion language, not the chrome. That insight shapes the entire ranking below.

Topic Axes

  1. Living graph canvas — node, edge, grouping, and layout craft in React Flow; the flagship surface.
  2. Real-time & motion — making the live-fold legible and alive: transitions, flow, replay, "now."
  3. App shell & navigation — information architecture, the operator screen set, command surfaces, moving between screens.
  4. Detail & data surfaces — node detail, gate cards, evidence, tables, metrics and forecast panels.
  5. Design language & identity — tokens, type, color, dark mode, density, and the distinctive-not-generic identity.

How to read this set. 01

Edges Carry the Factory

A1 · CanvasConf 85%Cx Medium

Put the distinctive identity where the research says it must live — the edges, not the chrome — and let the product's own name do the work. Dependency edges become the literal conveyor: directional flow animates only on active paths; a node's outgoing edges thicken and glow in proportion to starved_dependents, so a bottleneck visibly bulges with the work piling up behind it; blocked_by edges render as taut, dimmed tension that releases when satisfied. Nodes stay quiet status cards; the system's energy reads off the wires. Three independent frames (analogy/SCADA, constraint-flip, pain) converged here — the strongest signal in the set.

Basis
direct starved_dependents and blocked_by already exist on every node and are currently unvisualized. external SCADA P&ID mimic diagrams (animated flow on live lines, visible back-pressure at blockages) and Sankey flow-weighting. reasoned the product is literally named Conveyor; edge-as-flow is the one visual nobody else in the category builds, and it's data-true rather than decorative.
Rationale
This is the highest-leverage identity move available: it converts the strategic risk ("looks like every other dev tool") into a signature, and it answers the operator's first question — where is work piling up? — spatially, in ~100ms, instead of by clicking nodes.
Downsides
Edge weighting needs a sensible scale so a 40-waiter node doesn't dominate the canvas; over-animating flow re-introduces the "everything pulsing" noise the research warns against (mitigate: active-path only, motion off by default). Requires a custom React Flow edge renderer.
02

The Dark-Cockpit Color Law

A5 · IdentityConf 88%Cx Low

Borrow aviation's "dark cockpit" doctrine: when all is nominal, the panel is calm and unlit; saturated color appears only when something needs you, ranked by severity. Nominal states (running, ready_idle, done, skipped, parked) render in restrained monochrome; color is reserved for exceptions — failed = warning (red), blocked with high starvation = caution (amber), stalled = advisory. A persistent master-caution strip pins the single top-ranked exception so the operator never hunts for it, and a 2px ambient viewport border encodes overall health pre-attentively. This sharpens the generic "color = status" cliché into a law that does real work.

Basis
external aviation Crew Alerting System / "dark cockpit" doctrine (warning → caution → advisory tiers; master caution). direct the 8 states partition cleanly into nominal vs. exception; starved_dependents sets caution severity. reasoned "color only for status" is under-specified — equal color across 8 states is noise; severity-ranked scarcity makes one glance answer the unattended operator's real question, "does anything need me?"
Rationale
This is the difference between a pretty graph and a cockpit you trust to leave running. It is also the cheapest high-impact move in the set — primarily a token + rule decision, not a new component — and it directly serves the founder's walk-away thesis.
Downsides
Requires discipline to hold the line (every future screen must resist decorative color); colorblind-safe encoding must pair color with shape/icon; "what counts as an exception" needs a clear, stable definition so the panel doesn't cry wolf.
03

A Motion Grammar for State Change

A2 · MotionConf 82%Cx Medium

This is where premium and distinctiveness are actually manufactured — no whitespace dev-tool can copy motion. Define a fixed vocabulary of sub-300ms ease-out transitions, one signature per state change, animating only the node and edges a node:patch delta actually touched: a completing slice settles and releases its downstream edge; an unblocking slice's incoming edge fires and the node wakes (the RTS tech-tree "unlock pulse" — the proprietary signature); a failure fractures; a stalled node desaturates and cools; long-running nodes accrue a faint duration aura; and a completed node leaves a 300ms ghost at its old position so the eye keeps its place across re-layout. Shipped as one shared Motion variants object keyed by state — so the grammar is consistent everywhere and free on every new surface.

Basis
direct the server already streams node:patch deltas with old→new state — a free choreography source most tools waste by snapping. external Emil Kowalski / Linear motion craft ("<300ms, ease-out, animate only what changed"); RTS tech-tree unlock feedback; Material shared-element continuity. reasoned a watcher learns to read state peripherally by motion signature — meaning arrives before the label is read.
Rationale
Turns the real-time feed from a jarring snap into choreography, and makes the live-fold the thing that feels alive. Because it is one variants object, the motion identity propagates to list rows, badges, and toasts at zero per-screen cost.
Downsides
Easy to overdo — the grammar needs a strict budget and prefers-reduced-motion support; ghosting + re-layout interaction needs care so it reads as continuity, not lag; simultaneous deltas must batch gracefully.
04

The Design-System Spine

A5 · IdentityConf 90%Cx Medium

The compounding substrate that makes every future screen cheap and the whole app coherent — the highest-confidence direction in the set. Four pieces: (a) Semantic Status DNA — one token set (hue / saturation / luminance / motion) per state that every consumer reads, dark-mode-first, so adding a 9th state is one file edit; (b) the Node-Card Genome — a single <SliceCard> primitive at three scales (nano in the DAG, compact in lists, full in the detail rail) so the graph node is the list row is the panel header; (c) a reusable <LiveRail> typed-event-stream component the cockpit, gate audit, and evidence views all mount; (d) a ⌘K command palette with a handler-registration API so each new screen ships palette integration in ~15 lines.

Basis
direct the 8 states and the slice fields recur across every planned screen (runs, evidence, reviews, gates). external shadcn's "own the code" model + Radix primitives; cmdk command-palette as the standard operator-tool nav (Linear, Vercel); React Flow custom JSX nodes. reasoned making the DAG node and the list row the same component is the compounding insight — design effort spent once pays out on every surface.
Rationale
This is the pragmatic backbone that lets the ambitious identity ideas (1–3) actually reach the whole app instead of living only on one screen. It is the "build on what exists, compounds over time" move — exactly the leverage the founder optimizes for.
Downsides
Less visually exciting than the identity ideas, so easy to under-invest in early; over-abstracting too soon (before 2–3 real screens exist) risks the wrong primitives. Best built alongside the cockpit slice, extracted from it, not ahead of it.
05

Live Frontier + Epic Folding

A1 · CanvasConf 80%Cx Medium

Stop rendering the whole flat graph with equal weight — an autonomous run's interesting content is its moving edge, not its settled history. Render the execution frontier (the band of running + ready_idle slices) at full fidelity; fold the completed past into collapsed epic chips (each showing "7 / 12 done · 1 failed"); and foreshadow the near-future by lighting the slices whose blocked_by is about to clear. Semantic zoom is the filter — epics are React Flow sub-flow containers that expand on zoom-in, so a 3-slice smoke test and a 40-epic mega-plan use the same screen with no mode switch. The wavefront sweeps as a moving zone of attention; folding doubles as the scale answer (it keeps React Flow in its comfortable node range without a separate "1000-node mode").

Basis
direct epic_id defines the natural fold boundary; running/ready_idle/blocked_by define the frontier and its leading edge. external React Flow v12 sub-flows; Google-Maps semantic zoom; live-tail focus bands (Datadog). reasoned equal-weight rendering buries the 5 live nodes among 200 done ones; folding is both legibility and culling.
Rationale
Serves the glance-not-stare operator and removes the "overview vs. detail" toggle entirely (zoom is the toggle). It also quietly solves the React Flow performance ceiling without a special large-graph mode.
Downsides
"What counts as the frontier" needs a clear rule; auto-folding can hide something the operator wanted to watch (needs a pin/override); animated re-fold on every delta must be calm, not churny.
06

The Slice Dossier + Non-Vacuous Gate

A4 · DetailConf 78%Cx High

Break observe-only and close the loop from see the problem to fix the problem. Tapping a node opens not a read-only JSON panel but a dossier: the contract the slice promised, the evidence it produced, a failure fingerprint (a state-transition sparkline + the blocked_by chain), and — the centerpiece — the gate rendered as NASA mission-control's go / no-go poll. Each verification check is a named "controller" reporting GO / NO-GO / STANDBY with its evidence inline; the gate cannot pass while any check is NO-GO, and a single boolean checkmark becomes impossible. First-class actions live inline (approve / reject gate, retry, park, requeue). The graph stops being a museum behind glass and becomes where you drive the factory.

Basis
external NASA mission-control go/no-go polling (per-discipline controllers, unanimous GO required, named accountability). direct Conveyor's data model is already contract → evidence → review → gate — a dossier waiting for a surface; the prior ideation named "non-vacuous gate card" and "decide-on-the-node." reasoned a supervisor who can only watch must context-switch to a CLI to act, which guarantees the UI stays a toy.
Rationale
This renders Conveyor's entire trust thesis as UI — the product's hardest claim ("this work is verified") becomes legible and falsifiable at a glance. It is also what turns the cockpit from a viewer into an instrument.
Downsides
Highest build cost in the spine — needs the server to surface contract/evidence/gate detail and accept actions through the Channel, and inline actions raise auth/authority questions (the server must remain the decision authority; the UI only requests). Sequence it after the observe-only redesign lands.
07

The Run Wall / Flight-Strip Board

A3 · ShellConf 72%Cx Medium

The walk-away endgame ("hand Conveyor a big plan, it builds end-to-end") implies running several plans at once — which makes "one run at a time" the wrong default. Flip the run switcher into a fleet surface: a wall of living run-tiles (or air-traffic-control-style flight strips), each a compact miniature showing a per-state status bar, a pulse when a delta lands, and a flag when it needs a human — sorted by who needs attention. The full DAG cockpit is what you drill into from a tile. An unattended operator triages the whole factory at a glance before diving into one run.

Basis
external ATC flight-progress strips (one strip per concurrent track, ordered by attention) and small-multiples fleet boards (Grafana, Vercel deployments). direct "multiple runs; one viewed at a time"; the 8 states roll up to per-run health and exception counts. reasoned without a concurrent overview the operator can't answer "which run needs me?" factory-wide.
Rationale
Turns the cockpit from a single-run viewer into a factory-wide control surface — the unattended operator's real job is triaging N runs, which nothing today serves.
Downsides
Only earns its keep once multiple concurrent runs are common; risks duplicating the Needs-Me Inbox (#8) if both ship — pick the organizing metaphor deliberately rather than building both.
08

The Needs-Me Inbox

A3 · ShellConf 66%Cx Medium

The boldest reframe in the set — and a genuine strategic fork. It breaks the assumption that the graph is the centerpiece. In a working autonomous system, almost everything is fine almost all the time; the operator's scarce resource is the handful of moments that need a human (a gate waiting, a slice failed, a decision fork). So a persistent needs-me inbox becomes the spine of the app, and the beautiful DAG becomes the evidence surface you expand from an inbox item to see why. Supervise-by-exception, with the graph as the proof layer rather than the homepage. This is what makes the walk-away thesis genuinely operable — but it is in direct tension with making the live DAG the flagship (ideas 1–5), which is exactly why it deserves a deliberate decision.

Basis
reasoned in a working autonomous system the signal-to-noise of "watch the graph" is terrible — the scarce resource is human decisions, so the UI should be organized around those. external Linear inbox; PagerDuty/Opsgenie incident queue; GitHub required-reviews queue. direct prior capabilities "decide-on-the-node," "failure fingerprinting & BubbleUp," "non-vacuous gate card."
Rationale
It is the reframe that makes the founder's walk-away thesis operable — you can run unattended precisely because the few things that need you are pushed to a queue instead of hidden in a graph you must stare at.
Downsides
Directly competes with the "DAG as flagship/identity" thesis — committing here demotes the very surface ideas 1–5 invest in. Likely the right eventual shell but the wrong thing to lead with while the live graph is still the demo that sells the product. Treat as a deliberate fork, not a both-and.
09

The Walk-Away HUD

A4 · DetailConf 60%Cx High

An F1 pit wall continuously answers "given current pace, what's the projected outcome — and do we need to act?" Give each run a pit-wall HUD: projected finish, current pace vs. the plan's expected path, the single current bottleneck slice, and a prominent walk-away verdict — SAFE TO LEAVE / NEEDS YOU SOON / ACTING NOW, with the reason. Pair it with "Since You Last Looked": on re-entry, a recap of the deltas you missed ("4 slices done · 1 failed · gate now waiting") with the changed nodes pulsing before the view settles into live mode. Designs for the operator's actual relationship to an autonomous system — intermittent supervision — instead of pretending they stare all day.

Basis
external F1 pit-wall strategy screens (timing tower with deltas-to-reference, projected outcome, pit/stay call); macOS Time Machine / Slack unread-since-marker for the re-entry pattern. direct a plan-bearing system knows its own expected path; per-slice time-in-state and the delta stream are available. reasoned "is it safe to walk away" is the central unattended question with no quantitative home today.
Rationale
Converts a vague feeling ("seems fine?") into an explicit, evidence-backed go/no-go to stop watching — the Walk-Away Meter and Forecast delivered as one trustworthy HUD.
Downsides
The forecast/verdict is mostly a backend capability (a pace-vs-plan model Conveyor may not yet produce) — this is more than a frontend redesign, so confidence is lower and it depends on work outside this effort. The "since you last looked" half is cheaper and could ship first.
10

Time Transport

A2 · MotionConf 55%Cx High

Give the event-fold a DAW / video-editor transport. A time ruler with epics as stacked tracks, a "now" playhead riding the live edge, scrub to replay the run's state at any past instant, and a diff mode that pins two timestamps to show what changed between them. The node:patch stream is the timeline's event source; the playhead snaps to "live" or detaches to investigate history. This is the "what just broke and why" post-mortem made fluent — for the common case where you weren't watching when it happened.

Basis
external DAW / NLE transport + multitrack timeline (playhead, scrub, in/out diff regions — Ableton, Premiere); Temporal's Timeline/Compact replay views. direct the node:patch stream and epic_id tracks; prior capability "event-fold (live·replay·diff)." reasoned playhead-and-scrub is the most universally understood way to move through time over an event stream.
Rationale
Turns replay-and-diff from a feature into a craft-grade interaction operators already know how to use — essential for after-action review of an unattended run.
Downsides
The heaviest build in the set — requires the server to retain and serve full event history for time-travel reconstruction, which is real infrastructure beyond a frontend redesign. A "later" direction once the live cockpit and dossier exist.
