Living-Graph Cockpit

Verification overturned a scoping assumption

The light-control scope assumed pause / resume a run was a cheap, already-wired action. Fresh-eyes verification refuted it: there is no pause action, SerialDriver.resume! is crash-recovery only, and SerialDriver.run! is synchronous — there is no live run process to pause against. Pause is the one "control" that needs real new backend (an async run process + a :paused state). The other two controls — decide a parked gate and retry — are real and cheap. Scope corrected, not asserted.

This is largely an activation play, not new capture. The realtime substrate (transactional outbox → Conveyor.PubSub → LiveView), the rich data model, and even the run-reconstruction machinery already exist; what's missing is the projection. Eight of these ten directions are mostly wiring. The list clusters into a spine you build first (1–2), thesis-distinctive instruments (3–6, 8), operator leverage (7, 10), a fast delight win (4), and a later-when-volume-exists capability (9).

Grounding Context

Conveyor executes a frozen plan as a DAG of "slices" (tasks). A width-1 serial orchestrator (SerialDriver) walks the graph in dependency order; each slice runs through stations → agent sessions (implementer / reviewer / scout) → a patch + evidence → a 7-stage verification gate that emits pass/fail + a trust score. Everything is recorded append-only as LedgerEvents. North star: trustworthy autonomous verification when no human is watching — so the gate must be real, not theater.

What the UI is today. The web endpoint is live but thin: /runs (RunViewerLive) renders a nested-<div> Project→Plan→Epic→Slice hierarchy and dumps each event as pretty-printed JSON, re-fetching 17 resources and re-rendering the whole page on every event; /parked (ParkedQueueLive) is a read-only inbox of gate decisions that need a human but that the page cannot actually act on. There is no graph and no JS/asset pipeline anywhere (no esbuild, tailwind, or hooks).

What's already built (the activation surface). Realtime: EventOutboxRelay broadcasts {:ledger_event, …} over PubSub. Data: Plan→Epic→Slice with directed TaskDependency edges (:execution_hard); RunAttempt / StationRun (with heartbeat_at) / AgentSession; GateResult with 7 stages + a trust_score map (score / band / components / thresholds) + a false_negative flag; an append-only LedgerEvent stream; and reconstruction machinery (Conveyor.Replay, Conveyor.Planning.RunReconstruction) that already folds the ledger into ordered state.

Constraints that shape everything. ADR-09: PubSub is best-effort notification, not durable history — reconstruct from the ledger on mount, then subscribe for deltas, with idempotent handlers ordered by occurred_at (there is no integer sequence column). ADR-21: LiveView is a projection, not authority — every UI control must map to the same typed domain action the CLI uses; no UI-only state. Locked decisions: graph-as-home, render via LiveView + a JS graph hook (Cytoscape.js + elkjs recommended), observe + light control only.

Topic Axes

The subject was decomposed into five orthogonal surfaces; ideas are tagged with the one they most centrally target.

AGraph canvas & layout
How the DAG looks and navigates: layout, semantic zoom, node form, edges, legibility at 100s of nodes.
BLive execution overlay
How running state animates onto the graph: node-state encoding, the active-node detail rail, streaming, replay.
CVerification & trust surfacing
Making the gate cascade, trust score, and "why it failed" legible and non-vacuous — the anti-theater mission.
DGlanceability & attention-routing
The cockpit layer: at-a-glance health, "what needs me now," dark-cockpit, jump-to-problem.
ELight control & operator actions
The bounded mutations (decide parked gate, retry) surfaced on the node, mapped to typed domain actions.

C1

Render the stored edges — the living-graph substrate

Axis A · Graph canvas Confidence 92% Complexity Medium Role Spine · build first
Description

Replace the nested-<div> hierarchy with a Cytoscape.js canvas in a phx-hook (elkjs layered, left-to-right): Plan/Epic become compound parent containers, Slices are nodes, and the TaskDependency from→to edges — captured in the DB but never loaded by the web layer — finally get drawn. Width-1 execution means exactly one "live token" on the network at a time (a single-train read). Optionally overlay faint, clearly-labeled heuristic "latent-conflict" edges between slices that share a conflict_domain but have no hard dependency — a planning-blunder catcher.

Basis
direct
RunViewerLive renders a Project→Plan→Epic→Slice <div> hierarchy and load_projects/0 reads 17 resources but never TaskDependency (run_viewer_live.ex:482–518, 750–766); the edge is a directed from_slice→to_slice :execution_hard (task_dependency.ex:28–49); Slice.conflict_domains exists (slice.ex:126).
Rationale

The activation thesis in one move: the locked centerpiece (graph-as-home) literally doesn't exist, and the missing piece is data Conveyor already persists and WorkGraphBuilder already consumes. Every other direction in this document rides on this substrate.

Downsides

The real cost isn't backend — it's net-new frontend infra: with no asset pipeline today, this stands up the first hook and the first build. The latent-conflict overlay is a reasoned inference, not stored topology — keep it visually subordinate to the real edges so it never reads as a committed dependency.

C2

The event-fold engine — live · replay · diff

Axis B · Live overlay Confidence 88% Complexity Medium–High Role Spine · build first
Description

Today every ledger event triggers a full re-fetch of 17 resources and a whole-page re-render. Replace that with a pure fold over the LedgerEvent stream: seed once from durable history (ADR-09 reconstruct-then-subscribe), then apply each PubSub delta as a targeted push_event patch to the single node the event names — no re-fetch, no relayout. The same fold, parameterized by position, powers four surfaces: fold-to-head (live), fold-to-time-T (a DAW-style scrub playhead), fold-to-failure (debug freeze), and fold-since-last-seen ("what did I miss").

Basis
direct
handle_info({:ledger_event,_}) → assign_run_data → load_projects re-reads 17 resources on every event (run_viewer_live.ex:38–39, 862–868, 822); Conveyor.Replay.timeline! sorts events by occurred_at µs and RunReconstruction folds them into state (replay.ex:15, run_reconstruction.ex:50–66); LedgerEvent has no integer sequence column — ordering is occurred_at, dedup is idempotency_key (ledger_event.ex:23–28, 59).
external
DAW timeline + playhead; bv --robot-diff; ADR-09 reconstruct-then-subscribe.
Rationale

Fixes the crudest flaw in today's UI and turns the append-only ledger (a recording, not a dashboard) into navigable time — reusing reconstruction machinery already built. One fold retires several separate roadmap items (live, replay/time-travel, debug, catch-up).

Downsides

The scrubber must order by occurred_at, not a dense global integer (the only "sequence" is payload-local to the resume fold). A declarative event.type → node-patch registry is worth it but adds a small indirection layer. The eval-metrics facet is a downstream bonus, not load-bearing — don't let it gate the core.

C3

Honest node-state taxonomy

Axis B · Live overlay Confidence 88% Complexity Medium Role Thesis-distinctive
Description

Give the graph a vocabulary no other DAG tool has, because no other tool has the data. Running (the single active slice); Ready-but-idle (deps satisfied, waiting only because the driver is width-1 — surfaced as a "serial-tax: N could run now" counter); Blocked-by-unmet-dependency (computed from the edges, reading "blocked: SLICE-003 awaiting your gate" when the upstream is parked); Skipped-because-upstream-parked (Conveyor literally emits this); and Stalled (heartbeat gone cold or past the 1h cap — a dead worker, not a busy one). A park triggers an andon-style "skip-wave" that counts the starved dependents.

Basis
direct
SerialDriver :partial skip-and-continue walks a deterministic topo order and emits a skipped event carrying blockers (serial_driver.ex:74–76, 97–103, 199–217); StationRun.heartbeat_at + per-slice 1h cap; Slice.state has no blocked encoding, so it is computed from TaskDependency + the ready-set.
external
Airflow upstream_failed (the rare tool that separates blocked from not-queued); Toyota andon; CSS-only attribute diffs (never relayout on a status event).
Rationale

"Blocked-by-dep vs not-yet-queued" is the universal gap the field gets wrong; Conveyor uniquely has the edges to compute it and the parked-causality to make it actionable. The serial-tax counter doubles as the exact signal that tells you when parallelizing the driver would pay off.

Downsides

Requires the C1 substrate first. The stall threshold needs tuning — a false "stalled" on a legitimately slow station erodes trust, so honor the don't-fake-liveness rule and drive it from real heartbeat/cap data only.

C4

Run Radio + active-node detail rail

Axis B · Live overlay Confidence 85% Complexity Low–Medium Role Fast delight win
Description

Replace the per-event raw-JSON dump with a single live-scrolling plain-language commentary in the active node's detail rail: "Implementer wrote 3 files → tests 11/12 green → reviewer asked for rework on error handling → gate failed at DiffScope." It's a pure server-side projection — each LedgerEvent.type maps to one templated sentence. The rail blooms only for the active node (progressive disclosure: colour-glance → station/role/elapsed → full event drill-down), instead of today's ~10 always-expanded record lists per slice.

Basis
direct
RunViewerLive renders every event as Jason.encode!(payload, pretty: true) (run_viewer_live.ex:739, 897–901) and renders ~10 always-expanded per-slice record lists (590–727); the LedgerEvent.type vocabulary is already semantic (run.gated, gate.failed, agent.implemented…).
external
Kestra inline node logs; Ecological Interface Design (skill / rule / knowledge cognition registers).
Rationale

The single most-cited current-UI pain is "reads like a JSON dump." The data to narrate already exists; this is the cheapest move that most changes the felt experience — directly serving your "easier and more enjoyable to inspect" goal, and a strong early momentum win.

Downsides

Someone authors the type → prose templates (small, and ongoing as event types grow — pairs naturally with C2's registry). Narration can hide detail; keep the raw payload one click away for forensic moments.

C5

The non-vacuous gate card

Axis C · Verification Confidence 80% Complexity Medium Role Thesis-distinctive
Description

Bloom a node's gate into a NASA-style go/no-go cascade: the 7 stages each calling GO / NO-GO, with the first NO-GO auto-promoted as the headline "why" and the rest collapsed. Decompose the trust_score into SonarQube-style measured-value-vs-threshold rows (it persists components + thresholds the UI never shows). Mark a gate that passed only on retry as amber "low-confidence," not clean-green. And surface each stage's required? flag as a standing guardrail.

Basis
direct
GateResult.trust_score carries components / thresholds / policy_digest but the UI shows only band/score or just key:status with reasoning dropped (gate_result.ex:39–42; run_viewer_live.ex:930–936); RunReadModel.failing_stage already computes the first non-passed stage (run_read_model.ex:254–262); gate.ex:158 has a stage_passes_gate?(%StageResult{required?: false}) → true path.
external
NASA go/no-go launch poll; SonarQube quality-gate condition rows; flaky-test reporters' "low-confidence" tier.
Rationale

A non-vacuous gate must decompose into the conditions that produced it — the anti-theater mission that is Conveyor's north star. The reasoning is computed, stored, then string-mangled away; surfacing it makes a failure self-explanatory instead of a forensic exercise.

Downsides

Verified trim: the required?: false vacuity path is latent, not live — every production stage is required?: true today, so this is a forward guardrail (so a future non-required stage can't silently neuter the gate), not "here's the current hole." Don't oversell it as an active leak.

C6

Gate self-audit & trust calibration

Axis C · Verification Confidence 85% Complexity Medium Role Most thesis-aligned
Description

Turn the UI into an instrument that audits the verifier. A "Vacuity Radar" aggregates every GateResult.stages verdict across all runs and flags any of the 7 stages that has never once failed ("unproven"). A calibration view bins trust-band against actual downstream rework — if "high-trust" passes need rework as often as "low-trust" ones, the band is theater (a visibly flat reliability line). The canary false-negative rate (already computed) sits beside the score as the gate's own record of how often it let a known-bad through.

Basis
direct
GateResult carries passed + stages + false_negative (gate_result.ex:30–32); SwarmReadinessAudit computes canary_false_negative_rate = false_negatives/total exactly (swarm_readiness_audit.ex:197–200); GateHealth.false_negative_count exists and is already rendered as a bare string (gate_health.ex:34; run_viewer_live.ex:696).
external
SonarQube; flaky reporters; SLO burn-rate; the project's own "replay-fidelity / producer-vacuous" anti-pattern.
Rationale

The thesis made visible — "trustworthy verification when no one is watching." The UI can't make a fake gate real, but it makes fakeness legible and embarrassing, which compounds into pressure to fix the gate itself. The data is already on disk; this is pure activation and the most differentiated idea in the set.

Downsides

Verified caveat: the calibration signal only has teeth once false_negative actually gets set by the calibration loop. If it's perpetually nil/false, featuring it would itself be a producer-vacuous indicator — so add a "does this ever fire?" check before promoting it to a headline tile.

C7

The Walk-Away Meter

Axis D · Glanceability Confidence 80% Complexity Medium Role Strategically aligned
Description

Replace the header's vanity counts (Projects / Plans / Slices / Events) with the question the north star actually asks: a single top-left verdict — "nothing needs you for ~22 min — go" — computed from forward signals (no parked gates, no slice near its 1h cap, no recent low-band passes, next high-risk slice far off). The canvas stays dark (Airbus dark-cockpit: colour only for non-nominal); a persistent "N awaiting you" HUD pings and flies the viewport to the offending node. You don't watch the cockpit; it summons you.

Basis
direct
per-slice 1h / per-run 8h caps (a real countdown); slice.autonomy_level; ADR-21 projection parity; RunReadModel.verdict returns band/score; ParkedQueue's own moduledoc calls the parked queue "the only work the operator must triage."
external
Grafana top-left worst-of verdict; Airbus dark cockpit; SLO error-budget as a depletable quantity; RTS minimap-ping / jump-to-location.
Rationale

Every other cockpit optimizes "see everything"; this one optimizes "earn the right to leave." Permission to disengage is the literal endgame (run wider, unattended). It reframes the whole product from a screen you stare at into a system that interrupts you only when it must — and turns C3's states + C6's trust into one headline.

Downsides

Verified stretch: a typed CockpitStatus projection consumable by a second AI is net-new backend (a new resource/action), which collides with "minimal new backend." Keep the dark-cockpit glance + HUD (cheap, grounded); treat the machine-readable verdict as a later stretch.

C8

Forecast — the flight plan

Axis B · Live overlay Confidence 75% Complexity Medium–High Role Structurally uncopyable
Description

Because the driver is width-1 and walks a deterministic topological order, Conveyor already knows the exact sequence it will run next. Overlay a chess-engine-style "principal variation": the next K slices ghosted in execution order ahead of the live token, with per-slice ETAs from historical StationRun durations and a faint playhead ahead of "now." The instant a slice parks, the trail recomputes and the dependents it will skip ghost out — you see a stall's blast radius before it lands. Per-blocked-node "time-to-unblock" countdowns degrade honestly to "blocked by SLICE-003" when there's no history.

Basis
direct
SerialDriver walks do_topo in deterministic dependency order and never bails early (serial_driver.ex:97–103, 199–217); StationRun.started_at/completed_at give real durations; per-slice 1h cap bounds the estimate.
external
Chess-engine principal variation; weather nowcast time-to-impact; transit countdown boards.
Rationale

A monitor shows what is; trust-to-walk-away needs what's about to happen. The reframe from rear-view to windshield is something competitors structurally can't copy — their orchestrators aren't deterministic-serial. Pairs with C7: it's what lets you decide to stay-or-go before the high-risk moment.

Downsides

Verified: the order forecast is rock-solid; the time estimate is soft (width-1 + variable agent latency) — lead with order, hedge time. If the driver later goes parallel, "the next slice" becomes a wavefront and the exact-order framing needs a probabilistic version.

C9

Failure fingerprinting & BubbleUp

Axis D · Glanceability Confidence 72% Complexity Medium Role Later · scales with volume
Description

When ≥2 slices fail or park, auto-cluster them and surface the shared dimension Honeycomb-BubbleUp-style: "all 3 ran station apply_patch," "all share conflict_domain: auth," "all error_category: lease_lost." Fingerprint identical failures across runs into one issue with a count and first/last-seen (Sentry-style) instead of N identical reds; click it to highlight the implicated subgraph across every affected run.

Basis
direct
StationRun.error_category/error_message; Slice.conflict_domains/likely_files; AgentSession.role; GateResult.stages key; LedgerEvent entity links — the join keys all exist.
external
Honeycomb BubbleUp (rank which dimension differs in the failing population); Sentry fingerprint grouping (count + first/last-seen).
Rationale

A solo operator can't manually diff 50 red nodes. Correlated failures usually share one root; auto-finding it converts N investigations into one fix and feeds the "factory improves itself" loop — diagnostic value that grows, rather than drowns, as run count rises.

Downsides

Verified: value compounds with run volume — thin early for a solo operator with few failures, so best sequenced as a "later" capability once history exists. Fingerprint heuristics can over- or under-merge and need tuning.

C10

Decide-on-the-node control

Axis E · Light control Confidence 85% (core) Complexity Medium Role Closes the loop
Description

The whitespace nobody fills: surface the parked-gate decision (approve / keep-parked) and retry as primary actions on the node itself — wired to the same typed domain actions the CLI uses (ADR-21), with optimistic recolor that reconciles on the resulting ledger event. Pre-compute a recommendation and the single swing fact ("Recommend APPROVE — band B; only test_coverage is below threshold, by 0.03"). Order the parked queue by acuity (downstream slices starved × trust × age), not arrival. The operator becomes a judge ratifying pre-made decisions, not a pilot.

Basis
direct
ParkedQueueLive has no handle_event — the "needs-a-human" inbox literally can't act; the only web mutation is RunViewerLive's mark_external → HumanIntegration.record! (run_viewer_live.ex:43–44), a real typed action; ADR-21 maps controls to typed actions; trust_score components/thresholds give the swing fact; :partial gives downstream-starvation for acuity.
external
GitLab manual-gate inline play-button; ED triage by acuity; counterfactual "one fact that would flip it."
Rationale

Closes the loop the whole UI exists for — the one place a human is the bottleneck — and exposes that today's "needs-a-human" page can't take the human's decision. The actions already exist for the CLI, so the UI just calls them; pure activation.

Downsides

Verified — scope correction: the "pause/resume" control assumed during scoping does not exist (no pause action; resume! is crash-recovery; run! is synchronous with no live process to pause). Decide-gate and retry are cheap activation; pause is backend-gated (needs an async run process + a :paused state). The pre-armed "tripwire pause" variant is cut for the same reason — neuter it to a no-pause "page-me-when-this-parks" flag.
