Orientation — your bearings

The honest state: the serial loop is real and joined — M0–M3 are done (a real agent runs end-to-end through the production SerialDriver, rework-on-fail is ON, and a parked slice no longer halts the run). The trust gate (M4) is roughly half-activated: 7 of 14 stages wired, :abstain fires on real signals, calibration is real. The two big unbuilt fronts are M5 (autonomous decomposition) and M6 (long-horizon survivability).

Why you feel lost despite a strong roadmap: the ROADMAP already sequences the milestones, but it doesn't tell you which concrete moves are highest-leverage right now, or which "M6" items are actually preconditions sitting in the wrong milestone. That's what this document ranks.

Two debts loom over everything: the M3 exit evidence (a 3–8 slice live run, N consecutive, surviving an induced failure) is not yet banked, and value-over-bare-agent is unmeasured. Until both exist, every provisional exit-bar number is a hypothesis.

Grounding Context

Codebase Context

Conveyor is an AI-first "software factory" on the BEAM (Elixir/OTP) — 383 lib modules, 307 test files, 27 ADRs. Per the just-authored STRATEGY.md: the bet is that the BEAM's primitives (OTP supervision, the actor model, GenServers) inherit resilience and isolation rather than inventing them, with each task a headless CLI call carrying its own fresh context. The north star: hand Conveyor a large plan → it decomposes into a dependency-ordered work-graph of contract-bearing slices → agents build it unattended → the human reviews a morning digest and a small needs-judgment queue.

The candidates below were qualified against the ROADMAP's verified state (PR #19 merged) and re-checked in source by a fresh-context verifier. Load-bearing gaps confirmed in code: serial_driver.ex:163 hardcodes "status" => "matched"; zero Oban.insert in lib/; the run loop is in-memory Enum.reduce_while; decomposer.ex reads only List.first(requirements) and emits one slice with work_deps: []; corpus_pass_rate is a receive-only slot with no producer; ParkedQueueLive is display-only with no triage.

1

Event-sourced run ledger — the compound keystone

Axis A3Confidence 88%Complexity High
Description

Replace the in-memory reduce_while run loop with an append-only event-sourced ledger (Postgres, the first real Oban.insert): every slice attempt, gate verdict, and trust-score vector is committed before the next step. One structural change pays off four times — a crashed run replays from the last committed event (M6 durability), the recorded trace becomes the substrate the real replay-divergence producer diffs against (M4), the dormant Conductor/Oban substrate gets its first activation (the M7 seam), and the committed event stream is the eval dataset every metric needs.

Basis
direct: run loop is in-memory Enum.reduce_while (verified in agent_runner/pi.ex); zero Oban.insert/insert_all in lib/ (grep empty); Conductor children are no-op skeletons (conductor/child.ex init returns {:ok, Map.new(opts)}). All three confirmed by the fresh-context verifier.
Rationale

Every milestone from M4 to M7 depends on observing what actually happened in a run. The eval corpus isn't a separate project — it falls out of a durable ledger for free. No other single investment touches four milestones, and it's the move that turns "long run" into "survivable run."

Downsides

The heaviest item here — a real schema-design and state-machine effort, not a wiring task. Risk of over-building the ledger before the loop's event vocabulary has stabilized; mitigate by modeling only the events M4/M6 actually consume first.

2

Real replay-divergence producer — kill the hardcoded "matched"

Axis A1Confidence 95%Complexity Medium
Description

serial_driver.ex:163 emits "status" => "matched" unconditionally — every run report certifies perfect replay fidelity whether or not replay was exercised. Build the real producer: re-execute the recorded input trace through the eval ReplayEngine, diff the normalized event stream, and emit a divergence vector (which steps diverged, by how much) rather than a boolean — so replay_fidelity can actually go red and the gate can abstain on a non-deterministic agent. The divergence vector doubles as the substrate for time-travel debugging (A5).

Basis
direct: verifier confirmed lib/conveyor/planning/serial_driver.ex:163 hardcodes "status" => "matched" with no conditional branch (lines 155–168). The real divergence logic exists only in the eval path (cassette_bridge.ex), unwired to run reports. external: aviation flight-data-recorder pattern — store the full parameter trace, then compute divergence, never a bare pass/fail.
Rationale

This is the single most load-bearing fake signal in the codebase. While it's a string literal, M4's "gate honesty / zero false-pass" claim cannot honestly include replay as a tested dimension — the gate passes partly on a known lie. Smallest, highest-conviction honesty fix remaining.

Downsides

Needs a stable recorded trace to diff against — so it lands cleanly after (or alongside) idea 1's ledger. Defining "acceptable divergence" for a stochastic agent is a genuine design question, not a mechanical one.

3

Output-independent watchdog — close the silent-hang case

Axis A3Confidence 90%Complexity Low
Description

Verification sharpened this one: session_limits.ex does track max_wall_clock_ms and returns {:halt, …} on exceedance — but the check is event-driven, polled only when the agent emits an output line (pi.ex calls observe/2 per event). An agent that goes completely silent (network stall, infinite tool loop, hung subprocess) emits zero events and is therefore never reaped. Add a true OTP timer sentinel — Process.send_after(self(), :watchdog_fire, max_wall_clock_ms) per agent call — that fires independently of output and routes to rework/park via the existing AmendmentRouter path.

Basis
direct: verifier: session_limits.ex:55-56 halts on max_wall_clock_ms but only inside the per-event poll in pi.ex:224 — "a completely silent/hung agent that emits zero events would never trigger the halt." So the gap is specifically the silent-hang case, not a total absence of limits. external: a watchdog timer is the canonical embedded/OTP liveness primitive — bound by the clock, not by progress signals.
Rationale

The strategic call is "fully autonomous serial loop first." A loop that can hang forever on one silent 2 a.m. call is not autonomous — this is a precondition of the claim, currently misfiled as an M6 item. It's also the cheapest survivability primitive (one GenServer, ~80 lines, no new deps) and the foundation the M6 anti-thrash sentinel later plugs into.

Downsides

Picking the timeout value is a calibration problem — too tight reaps legitimately slow tool calls, too loose defeats the purpose. Needs a per-tier budget, not one global constant.

4

Real decomposer — a computed dep-graph from interface bindings (M5)

Axis A2Confidence 92%Complexity High
Description

Today Decomposer.propose reads List.first(requirements) and emits exactly one hardcoded slice with work_deps: [] — the "dependency-ordered work-graph" north star does not exist. Build the real decomposer: drive PlanFoundry/CodexDrafter from mix conveyor.author "intent" to produce N contract-bearing slices, compute real work_dependencies from interface bindings (SliceDependency/InterfaceGraph provider→consumer edges), validate the DAG is acyclic, and fire the ADR-27 Interrogator on genuine ambiguity. Treat it as mise en place: the line refuses to start until every slice has its contracts stationed.

Basis
direct: verifier confirmed decomposer.ex calls List.first(requirements) (line 30), builds one slice with work_deps: [] (lines 38–46), never iterates requirements, never computes dependencies. SliceDependency and work_graph_lowering.ex:38 already define the canonical target format. reasoned: the computed dep-graph IS the parallel execution plan — building it for M5 is the same artifact M7 needs, so it compounds forward.
Rationale

M5 is explicitly a precondition of the §4 exit bar — without autonomous decomposition, mix conveyor.run requires a hand-authored plan, meaning the operator still does the most expensive cognitive work the system exists to eliminate. This is the line between "a factory" and "a fancy task-runner."

Downsides

Highest-uncertainty quality bet in the roadmap (ROADMAP R4) — decomposition quality is the load-bearing artifact for the whole dream and will need iteration. Large; best attempted once ideas 1–3 make runs durable and honest enough to trust the feedback.

5

Multi-target corpus + per-stage defect-density map

Axis A4Confidence 85%Complexity Medium
Description

Every exit-bar metric today is measured on one target: Beads-Insight, a pure-logic greenfield Python repo — the easiest possible autonomous-coding task. Build a corpus harness of ≥3 distinct archetypes (pure-logic, a stateful CLI/service, a schema-driven library), run the MutantGauntlet canary corpus across the grid, and produce a per-stage × per-archetype escape-density map. A single hot cell tells you a gate stage is structurally blind to a defect class, not merely unlucky — and the run produces the real corpus_pass_rate signal the trust path is already wired to receive.

Basis
direct: verifier confirmed corpus_pass_rate exists in gate/trust_evidence.ex (lines 58, 80) and is consumed in trust_score.ex, but is read-only via Map.get(output, "corpus_pass_rate") with no producer anywhere in lib/. ROADMAP §4: every bar metric is measured on the single Beads-Insight target; ≥3 targets would "materially de-risk over-fitting." external: semiconductor yield engineering — qualify a process on a defect-density map across many wafers, never a single one.
Rationale

A single data point is not a measurement. The "≥70% first-pass gate success" target measured on one trivial target could be a fluke of its structure; thresholds calibrated there may be systematically too loose for anything with I/O or state. Discovering that now is cheap; discovering it mid-medium-plan is a crisis. Also the first proof Conveyor works on something other than easy Python — load-bearing for the open-source story.

Downsides

Sourcing/authoring good greenfield targets is real curation work, and every added target multiplies agent-token cost per calibration run — a scheduling input the ROADMAP §6 already flags.

6

Bank the M3 exit run + a lift-over-bare-agent harness

Axis A4Confidence 88%Complexity Medium
Description

The M3 exit evidence (a real 3–8 slice live run, N consecutive, surviving an induced slice failure) is not yet banked, and value-over-bare-agent is unmeasured. Build a mix conveyor.exit_bar_run harness that executes the prescribed sequence, runs a bare-agent control arm on the same plan, and emits a signed evidence-ledger artifact measuring the only metric that justifies the product: defects-caught and honest-abstention (verifier outputs), not pass@1. Exit-bar clearance becomes a green/red assertion you can re-run, not a manual ceremony.

Basis
direct: verifier confirmed zero matches for exit_bar, bare_agent, control_arm, or consecutive_run anywhere in lib/ or test/ — no such harness exists. ROADMAP: M3 exit pending; pass@1 was the wrong metric (n=3 trivial, both arms passed 3/3 → no signal). reasoned: the whole strategy rests on "trustworthy completion + defects-caught beats a bare agent" — until that's measured once, it's a hypothesis, and it only becomes measurable once M4's gate is active.
Rationale

This is the evidence debt that makes everything else honest. Automating it forces the exit criteria to be machine-checkable (surfacing ambiguity before it becomes a dispute) and converts "have we cleared M3?" from an open question into an assertion. It's also the first time you'd prove Conveyor catches a defect a bare agent would have merged.

Downsides

Partly gated on M4 being active enough to emit real defect-caught/abstain signals, and on idea 5's corpus to avoid measuring lift on one trivial task. Running a control arm doubles token cost per evaluation.

7

Decision-forcing morning digest & parked-queue triage

Axis A5Confidence 78%Complexity Medium
Description

ParkedQueueLive exists but is display-only — it lists abstained runs least-trusted-first with no triage. Turn the morning digest into a decision-forcing dossier: each parked slice gets an auto-assigned triage tag (red = real defect signal, needs you now; yellow = retry-eligible; green = cosmetic/auto-resolvable; black = unrecoverable), with the specific gate finding and attempt trace pre-staged so a judgment takes <60 seconds. Capture the human's verdict on each item as a structured label — which becomes the ground-truth signal for the next round of gate calibration (the A4 flywheel).

Basis
direct: verifier confirmed ConveyorWeb.ParkedQueueLive is a real LiveView (sorts by trust, refreshes via PubSub) but has "no auto-triage tagging, no digest generation, no filtering UI, no triage action callbacks"; no triage_tag symbol exists in lib/. external: emergency-department START triage — a fixed protocol lets a small staff sort a flood by attention-given-survivability, not arrival order.
Rationale

Operator throughput on the parked queue is a hard ceiling on unattended scale — the exit bar's "≤N judgment-items/day for one reviewer" cap is a wish without triage and an enforced budget with it. If each item takes 5 minutes of log archaeology, N is small; a pre-staged dossier compresses that 5–10×. This is the primary interface between the factory and the human.

Downsides

Lower confidence: its value depends on real overnight runs existing to triage, so it trails ideas 1, 4, and 6. Auto-triage tag accuracy is itself a quality problem — a wrong "green" tag is exactly the silent failure the gate exists to prevent.

5 Provenance-attestation chain-of-custody stage Subject-aligned and valuable for the M8 self-hosting capstone's audit trail, but below the "what next" horizon — it serves a milestone gated behind the entire exit bar.
6 Contract-synthesis assistant (help the human author contracts) Genuinely interesting and challenges the "human writes the contracts" assumption — but better developed as a ce-brainstorm variant of M5 than built standalone; it overlaps idea 4's decomposition surface.
